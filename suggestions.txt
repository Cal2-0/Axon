# ═══════════════════════════════════════════════════════════════════════
# PILLAR I: CHAIN OF CUSTODY & INSTITUTIONAL CASE MANAGEMENT
# ═══════════════════════════════════════════════════════════════════════
# "If it can't survive a defense attorney's cross-examination, it's not evidence."

### 1. Persistent Case Workspaces
# Status: [IN PROGRESS] — basic Cases page exists
# Every investigation must live inside a named, numbered Case (e.g., Case #2026-ETH-041).
# Analysts group dozens of wallets, contracts, bulk scans, and OSINT reports
# into a single unified workspace that persists across sessions, browser refreshes,
# and server restarts. Cases must support:
#   - Case metadata (title, description, assigned analysts, priority level, status)
#   - Tagging and categorization (ransomware, pig butchering, rug pull, etc.)
#   - Case timeline showing every action taken chronologically
#   - Linking related cases (e.g., same threat actor across multiple incidents)

### 2. Cryptographic Evidence Hashing & Chain of Custody
# Status: [NOT STARTED]
# Every scan result, PDF export, and graph snapshot must be cryptographically
# hashed (SHA-256) at creation time with an exact UTC timestamp. This creates
# an immutable evidence chain proving:
#   - WHEN the blockchain data was pulled
#   - WHAT the exact data was at that moment
#   - WHO pulled it (analyst ID)
#   - That it has NOT been tampered with since extraction
# The hash manifest should be exportable as a standalone "Chain of Custody Certificate"
# that accompanies every evidence package submitted to prosecutors.

### 3. Role-Based Access Control (RBAC) & Audit Trails
# Status: [NOT STARTED]
# A Lab Commander must audit their own team. Required roles:
#   - Admin: Full system access, user management, DB configuration
#   - Lead Investigator: Case creation, analyst assignment, report approval
#   - Analyst: Scanning, analysis, note-taking within assigned cases only
#   - Read-Only / Auditor: Can view cases and logs but cannot modify anything
# Every action (search query, scan trigger, export, note edit) must be logged
# in a tamper-proof internal audit trail. This prevents insider tipping,
# evidence mishandling, and ensures ISO 27037 compliance for digital evidence.

### 4. Dynamic Cloud Logs & Persistence (Supabase Integration)
# Status: [NOT STARTED]
# Transition from ephemeral local JSON/SQLite logs to a persistent, globally
# accessible Supabase backend. Benefits:
#   - Investigation data survives server restarts and redeployments
#   - Multiple forensic labs can share real-time threat data
#   - Row-level security ensures analysts only see their assigned cases
#   - Real-time subscriptions for live collaboration between analysts
#   - Automatic backup and disaster recovery

### 5. Evidence Intake & Seizure Documentation
# Status: [NOT STARTED]
# Before analysis begins, a forensic lab must document HOW the crypto evidence
# was obtained. AXON needs an "Evidence Intake Form" that captures:
#   - Source of the wallet/address (seized device, informant tip, subpoena response)
#   - Legal authority (warrant number, court order reference)
#   - Physical evidence chain (which device, which app, screenshot of seed phrase)
#   - Hash of original intake data
# This is the FIRST thing a defense attorney will challenge.

# PILLAR II: ADVANCED TRACING & GRAPH ANALYTICS
# ═══════════════════════════════════════════════════════════════════════
# "Show me where the money went. Every hop. Every split. Every dead end."

### 6. Automated "Follow-The-Money" Tracing Engine
# Status: [NOT STARTED]
# The investigator sets parameters:
#   "Trace all outbound funds > 1 ETH, up to 10 hops, stop at Exchange or Mixer."
# AXON recursively crawls the blockchain in the background and generates
# the exact flow path:
#   Hacker → Intermediary 1 → Intermediary 2 → Tornado Cash → Intermediary 3 → Binance Deposit
# Each hop is annotated with: amount, timestamp, gas fee, and entity label.
# The trace must handle splitting (1 wallet → 5 peel chains) and merging
# (5 wallets → 1 consolidation address).

### 7. Entity Evidence Board (Maltego-Style Interactive Canvas)
# Status: [NOT STARTED]
# A drag-and-drop canvas where analysts can:
#   - Push any scanned wallet/contract onto the board
#   - Manually add notes, labels, and color-coded groups
#   - Let AXON auto-discover hidden links between entities
#   - Draw the "Mastermind" wallet orchestrating a campaign
#   - Real-time multiplayer collaboration (Figma for forensics)
#   - Export the board as a high-resolution image for court presentations

### 8. Cross-Chain & Bridge Analytics
# Status: [IN PROGRESS] — basic cross-chain module exists
# Hackers rarely stay on one chain. AXON must track assets as they jump:
#   Ethereum → (Thorchain) → Bitcoin → (Ren) → Tron → (Circle Bridge) → Solana
# Required capabilities:
#   - Detect bridge contract interactions automatically
#   - Correlate the wrapped asset on the destination chain
#   - Support: Ethereum, Bitcoin, Tron, Solana, BSC, Polygon, Arbitrum, Optimism, Base
#   - Flag privacy chain off-ramps (Monero, Zcash) as "trail gone cold"

### 9. Mempool & Time-Series Activity Analysis
# Status: [NOT STARTED]
# Investigators need temporal analysis:
#   - Gantt-style timeline scrubber showing when a wallet is active
#   - Heatmap of transaction frequency by hour-of-day and day-of-week
#     (deduce geographic timezone of the operator)
#   - Mempool forensics: analyze pending/failed transactions for intent
#   - MEV interaction detection (sandwich attacks, frontrunning)
#   - Dormancy gap visualization ("wallet slept for 2 years, then moved 500 ETH")

### 10. Sankey Flow Diagrams
# Status: [NOT STARTED]
# The D3 force graph is great for topology but poor for showing volume.
# Add a toggle to switch to a Sankey diagram that visually represents:
#   - The VOLUME of funds flowing from source to destination
#   - Splitting into peel chains
#   - Aggregating into exchange deposits
# This is the gold standard for "follow the money" presentations in court.

### 13. Dark Web & Telegram Intelligence Scraping
# Status: [NOT STARTED]
# Deep OSINT layer:
#   - Ransomware negotiation chats (Conti leaks, LockBit panels)
#   - Darknet Market vendor profiles and withdrawal addresses
#   - Telegram Drainer-as-a-Service group monitoring
#   - Paste sites (Pastebin, Ghostbin) for leaked address lists
#   - Russian/Chinese hacker forum monitoring (XSS, Exploit.in, BreachForums)

### 14. IP Telemetry & Off-Chain Correlation
# Status: [NOT STARTED]
# The holy grail of physical attribution:
#   - Integrate with proprietary RPC nodes to capture IP addresses
#     broadcasting transactions to the Ethereum mempool
#   - Correlate IP → ISP → Geographic location → Physical person
#   - Cross-reference with VPN/Tor exit node databases to identify
#     whether the broadcaster is using anonymization
#   - This is how the FBI caught the Colonial Pipeline attackers

### 15. Device Forensics Integration
# Status: [NOT STARTED]
# When a physical device is seized (phone, laptop, hardware wallet),
# AXON should accept imports from:
#   - Cellebrite UFED extraction reports (XML/UFDR format)
#   - MetaMask local storage dumps (Chrome extension data)
#   - Hardware wallet transaction logs (Ledger, Trezor)
#   - Browser history showing DeFi/exchange interactions
#   - Seed phrase / private key recovery artifacts
# This bridges the gap between physical device forensics and blockchain analysis.

### 16. AI Dynamic Learning System & Auto-Updater
# Status: [NOT STARTED]
# When AXON's behavioral engine or an analyst confirms a suspicious wallet:
#   - The AI extracts the behavioral fingerprint (tx rhythm, fan-out pattern,
#     denomination clustering, dormancy profile)
#   - Auto-updates the axon_intel.db corpus with the new signature
#   - All future scans benefit from this learned pattern
#   - New exploit techniques (e.g., a novel bridging pattern) are detected
#     and catalogued WITHOUT manual intervention
# This transforms AXON from a static tool into a living intelligence engine.

### 17. Behavioral Clustering & Campaign Detection
# Status: [NOT STARTED]
# When investigating a massive exploit (e.g., Lazarus Group):
#   - AXON auto-clusters wallets by behavioral fingerprint similarity
#   - If 5 wallets show identical peel-chain timing and round-trip patterns,
#     cluster them as "Likely Single Operator Campaign"
#   - This attribution happens BEFORE any database lookup
#   - Competitive moat: Chainalysis relies on label lookup; AXON uses behavior

### 18. Natural Language Case Querying (AI Copilot)
# Status: [NOT STARTED]
# An investigator should be able to type in plain English:
#   "Show me all wallets in Case #041 that interacted with Tornado Cash
#    after January 2024 and moved more than 10 ETH"
# AXON's AI translates this into structured queries against the case database
# and returns results instantly. This eliminates the SQL/filter learning curve
# for non-technical law enforcement officers.

### 19. Predictive Risk Modeling
# Status: [NOT STARTED]
# Based on historical patterns of known exploit wallets:
#   - Predict the NEXT likely action of a suspicious wallet
#   - "This wallet's pattern matches 87% of known rug-pull operators.
#     Based on historical data, the operator is likely to drain liquidity
#     within the next 48-72 hours."
#   - Alert the investigator to act preemptively

### 20. Multi-Language AI Narrative Generation
# Status: [NOT STARTED]

#   - Adjust technical depth based on audience (expert vs. jury)
